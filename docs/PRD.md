## 2. 产品定义

### 2.1 核心叙事

> **"大学生 + AI = 数控老师傅七成功力"**

让一个刚毕业的大学生拿着这个产品，就能独立完成数控编程、排故和参数调优。把老师傅脑子里的经验，变成年轻人手机里能用的工具。

### 2.2 产品目标

| 编号 | 目标 | 衡量标准 |
|------|------|----------|
| G1 | **降低编程门槛**：让仅有机床操作基础的大专/本科毕业生能够独立完成数控编程 | 用户可在 15 分钟内完成从图纸到可用 G 代码的全流程 |
| G2 | **减少撞刀事故**：通过多层安全校验，将因编程错误导致的撞刀风险降至最低 | A 类阻断错误零遗漏；B 类警告 100% 覆盖已知危险模式 |
| G3 | **提升排故效率**：通过 AI 问答在 1 分钟内给出故障诊断建议，替代老师傅"凭经验猜" | 常见故障（振纹/烧刀/尺寸超差）首条诊断命中率 ≥ 70% |

### 2.3 用户故事

| 编号 | 角色 | 需求 | 价值 |
|------|------|------|------|
| US1 | 刚入职的 CNC 操作工 | 上传一张零件图纸，选择机床型号，自动生成能直接用的 G 代码 | 不用自己写代码，不怕写错撞刀 |
| US2 | 车间操机员 | 加工时遇到振纹，拍照描述问题，AI 告诉我该调哪个参数 | 不用等老师傅来救场，自己就能排查 |
| US3 | 工艺准备人员 | 输入材料（如 45#钢）、工序（外圆粗车）、刀具（硬质合金），秒出切削参数 | 不用翻手册翻半天，手机一点就有推荐值 |
| US4 | 车间主任/老板 | 看新人产出和错误率有没有改善 | 降低对高薪老师傅的依赖，新人上手更快 |
| US5 | 不同机床的操机员 | 同一段刀路，选不同机床自动适配对应 G 代码方言 | FANUC/西门子/Haas/广数之间切换不用手动改代码 |

## 3. 用户画像与场景

### 3.1 主要用户：大专/本科毕业生操作工

- **年龄**：22–26 岁
- **技能**：有机床操作基础（会装夹、对刀、按循环启动），但缺乏独立编程和故障诊断能力
- **痛点**：
  - 学校学的 FANUC，到了厂里是 Siemens，代码不会写
  - 出了问题（振纹、尺寸不对）不知道从哪查起
  - 怕写错代码撞刀，不敢自己动手
- **场景**：车间现场，手机操作为主（一手油，希望流程越短越好）

### 3.2 次要用户：工厂老板 / 车间主任

- **关注点**：人工成本、废品率、新人上手周期
- **痛点**：
  - 老师傅难招、薪资高、流失风险大
  - 新人培养周期长（3–6 个月才能独立编程）
  - 编程错误导致的撞刀一次损失数千至上万元

### 3.3 典型使用场景

| 场景 | 描述 | 触点 |
|------|------|------|
| 日常编程 | 拿到新图纸 → 拍照上传 → 确认解析结果 → 选机床 → 生成 G 代码 → 拷到机床上 | G 代码生成全流程 |
| 现场排故 | 加工异常 → 打开对话问答 → 描述问题/拍照 → 获取诊断建议 → 调整参数 | AI 对话问答 |
| 工艺准备 | 新零件新材质 → 打开参数速查 → 选材+工序+刀具 → 获得推荐参数 → 录入机床 | 参数速查 |
| 多机床切换 | 同一零件在不同机床上加工 → 切换机床型号 → 自动适配 G 代码方言 | 后处理模板化 |

### 3.4 用户任务流

> **设计原则：** 交互说明应以"用户完成任务需要多少步"而非"系统提供哪些功能"定义。以下任务流展示真实步数（含分支和重试路径），作为第 4 章功能设计和第 13 章交互说明的约束。

#### 3.4.1 任务流一：日常编程（图纸 → G 代码）

完整步数图（含分支和重试路径）：

```
1. 进入 G 代码 Tab
   └→ 1a. 已有历史记录 → 可选择加载历史档案（跳 1b）
2. 上传毛坯图纸
   ├→ 2a. 质量门禁不通过 → 重拍/重新上传（重试）
   └→ 2b. 拍照后本地质检失败 → 本地预览提示后重拍（2 步反馈）
3. 上传成品图纸（同上分支）
4. 选择原料材质（下拉单选）
5. 选择机床品牌型号（下拉单选）
   └→ 4-5 完成后自动跳转下一步
6. AI 解析两张图纸
   ├→ 6a. 低置信度项（<60%）→ 逐项编辑确认
   ├→ 6b. 中置信度项（60-85%）→ 用户可选择性确认
   └→ 6c. 高置信度项（≥85%）→ 自动通过
7. 用户确认解析结果 → 点击"确认无误，生成 G 代码"
8. AI 生成 G 代码 → 安全校验引擎检查
   ├→ 8a. A 类阻断错误 → 回退 AI 重生成（最多 2 次）
   │   └→ 2 次仍不通过 → 展示代码 + 错误详情 + 建议手动修改
   ├→ 8b. B 类警告 → 展示警告清单，用户逐项勾选确认
   └→ 8c. C 类提醒 → 代码中标黄高亮
9. 用户确认安全须知 → 勾选 B 类警告 + 阅读免责声明
10. 复制 G 代码
11.（可选）进入首件检验
    ├→ 11a. 检验合格 → 自动归档加工记录
    └→ 11b. 检验不合格 → AI 分析原因 → 反馈修正（回到步骤 8）
```

**真实步数：** 11 步（主线）/ 13+ 步（含分支重试）

**优化目标：** 二期通过加载历史档案可将主线压缩至 6-8 步。△当前 PRD"核心流程≤6 步"承诺不准确，以上为实际步数。

#### 3.4.2 任务流二：现场排故

```
1. 进入对话问答 Tab
2. 拍照/描述问题（文字 + 可选图片）
   ├→ 2a. 语音输入（P1 功能）
3. AI 通过 RAG 检索知识库生成回答（≤3s 首字响应）
4. 用户按照建议调整参数
5.（可选）验证效果后回到对话反馈结果
```

**真实步数：** 4 步（主线）/ 5 步（含反馈）

#### 3.4.3 任务流三：首件检验

```
1. 完成 G 代码生成 → 点击"开始首件检验"
2. 查看 AI 自动生成的检验清单（基于图纸解析结果）
3. 逐项测量并填入实测值
   ├→ 3a. 合格 → 绿色标记
   ├→ 3b. 不合格 → 红色标记 + AI 可能原因分析
   └→ 3c. 不会量 → 点击"不会量？问AI"跳转对话（自动带工件上下文）
4. 确认所有项合格 → 提交保存加工记录
```

**真实步数：** 4 步（主线）/ 5+ 步（含不合格项修正）

#### 3.4.4 修正说明

当前 PRD 第 9 章"核心流程（图纸→G 代码）在手机端可完成，交互步骤 ≤ 6 步"为虚假承诺。以上任务流分析显示图纸→G 代码主线为 11 步（含 AI 处理时间）。**建议：** 将第 9 章步数要求修正为"日常编程主线 ≤ 12 步，排故主线 ≤ 5 步"，二期通过历史档案加载和离线缓存将日常编程压缩至 ≤ 8 步。

## 4. 核心功能详解

操作端共 2 个 Tab：「对话问答」「G代码」。参数速查不作为独立页面，其数据能力融入对话问答和G代码生成流程中。

---

### 功能 1：AI 对话问答（含报警速查 / 切削参数问答）

#### 4.1.1 功能概述

基于行业知识库的实时问答系统。用户以自然语言描述问题（支持文字 + 图片），系统通过 RAG（检索增强生成）检索相关知识片段，结合大模型生成带来源引用的回答。同时覆盖机床报警代码速查、切削参数问答等高频场景。

#### 4.1.2 知识库覆盖范围

| 领域 | 内容 |
|------|------|
| **材料库** | 碳钢（20#/45#/40Cr）、不锈钢（304/316）、铸铁（HT200/HT250）、铝合金（6061/7075）、铜合金、钛合金 |
| **故障库** | 振纹（刀杆悬伸过长/转速-进给不匹配）、烧刀（冷却不足/切速过高）、尺寸超差（刀具磨损/让刀/热变形）、表面粗糙度不达标、断屑不良、锥度异常 |
| **工艺库** | 车削/铣削/钻削/镗削/攻丝的典型工艺参数范围、装夹方式选择原则 |
| **报警码库** | FANUC PS/AL系列、Siemens AL系列、Haas ALARM系列、GSK报警码 — 每条包含报警含义、常见原因、解决方案 |
| **切削参数** | 各材料×工序×刀具的推荐切削参数（切削速度、进给、切深、冷却建议），不单独展示，作为AI回答的底层数据 |

#### 4.1.3 跨Tab上下文共享

对话问答与G代码Tab之间共享加工上下文。当用户从G代码流程中点击"不会量？问AI"跳转到对话时，当前工件信息（材料、机床、零件类型、当前工序）自动注入对话上下文，AI回答时可引用这些信息，避免用户重复描述。

#### 4.1.3 交互流程

```
用户输入（文字+可选图片）→ 意图识别 → 知识检索（RAG）→ LLM 生成回答 → 带引用展示
```

#### 4.1.4 功能需求

- **FR-1.1**：支持多轮对话，保持上下文记忆
- **FR-1.2**：回答必须附带知识来源引用（文档名/段落）
- **FR-1.3**：支持上传图片辅助描述问题（如工件表面照片）
- **FR-1.4**：响应时间应在 3 秒内
- **FR-1.5**：支持对话历史查看（最近 50 条）

#### 4.1.5 优先级

| 需求 | 优先级 |
|------|--------|
| 文字问答 + RAG 检索 | P0 |
| 知识来源引用展示 | P0 |
| 图片上传辅助 | P1 |
| 对话历史 | P1 |
| 语音输入 | P1 — 第 3 章场景明确写"一手油，希望流程越短越好"，语音在车间是刚需非增值功能 |
| 离线缓存常见答案 | P2 |

### 功能 2：G 代码生成 + 首件检验 + 刀具追踪（核心全流程）

> **设计原则**：参数速查不做独立Tab。切削参数作为底层数据融入AI对话问答（用户直接问就回答）和G代码生成（AI自动选取最佳参数填入代码）。操作端只保留「对话问答」和「G代码」两个Tab，界面更干净，用户路径更短。

#### 4.3.1 功能概述

这是产品的核心差异化功能。完整流程：

```
上传毛坯图纸 + 上传成品图纸 → 质量门禁检查（两张分别校验）→ 
选原料材质 + 选机床品牌型号（纯下拉，后台维护）→ 
AI 分别解析两张图纸 → 比对毛坯与成品计算加工余量 → 
双图纸预览 + 双解析卡片展示（支持逐项编辑）→ 用户确认 →
后处理模板注入 → AI 生成 G 代码 → 安全校验 → 装夹方案 + 刀路模拟 → 
风险确认 → 输出代码
```

#### 4.3.2 子功能详解

##### 子功能 3A：双图纸上传 + 质量门禁

**上传要求：**

| 图纸类型 | 说明 | 示例 |
|----------|------|------|
| **毛坯图纸** | 原材料尺寸图、锻件图、铸件图，体现"加工前"的状态 | 棒料规格、铸件毛坯图 |
| **成品图纸** | 加工后零件图，带尺寸公差、粗糙度、形位公差等完整标注 | 标准机械零件图 |

两张图纸均需通过质量门禁。系统自动比对毛坯和成品尺寸，计算加工余量（如毛坯D50→成品D45，单边余量2.5mm）。

**上传方式：** 支持本地文件上传（jpg/png/pdf）和拍照上传（调用手机相机）。

**质量门禁检查（双阶段）：**

**阶段一：本地实时质检**（拍照后立即执行，不上传服务器）

| 检查项 | 门限值 | 不通过处理 |
|--------|--------|-----------|
| 分辨率 | ≥ 800×600 px | 预览页实时叠加"请调整：当前分辨率不足"提示，给出参考示例 |
| 倾斜角度 | ≤ 15° | 预览页实时显示水平参考线 + "请摆正后重拍"提示 |
| 清晰度（初步） | 边缘检测粗略估算 | 显示"建议确认光线充足" |

本地质检在 500ms 内完成反馈，用户确认预览 OK 后再上传高清图做服务器精细分析。失败反馈从"5 步后"提前到"2 步后"。

**阶段二：服务端精细校验**（上传后执行）

| 检查项 | 门限值 | 不通过处理 |
|--------|--------|-----------|
| 清晰度 | Laplacian 方差 ≥ 100 | 提示光线不足/对焦不清，建议重拍 |
| 文件类型 | jpg/png/pdf | 不支持的类型给出明确说明 |
| 文件大小 | ≤ 10MB | 提示压缩或裁剪 |

##### 子功能 3B：双图纸 AI 解析 + 加工余量计算

AI 分别解析毛坯图纸和成品图纸，提取结构化信息，并自动比对计算加工余量。

**毛坯图纸提取项：**

| 提取项 | 说明 | 置信度分级 |
|--------|------|-----------|
| 毛坯外径/长宽 | 原材料关键尺寸 | 高（≥85%）→ 绿色；中（60-85%）→ 黄色；低（<60%）→ 红色 |
| 毛坯长度 | 棒料/锻件总长 | 同上 |
| 毛坯类型 | 棒料/锻件/铸件/板料 | AI推测，中低置信度为主 |

**成品图纸提取项：**

| 提取项 | 说明 | 置信度分级 |
|--------|------|-----------|
| 零件类型 | 轴类/盘类/套类/支架类 | 同上 |
| 关键尺寸 | 外径、内径、长度、孔位等 | 同上 |
| 公差标注 | φ50±0.02 / φ30H7 等 | 同上 |
| 表面粗糙度 | Ra 0.8 / Ra 3.2 等 | 同上 |
| 形位公差 | 同轴度/垂直度/跳动 | 中低置信度为主 |
| 螺纹/倒角/槽等特征 | M20×2.0 / C1 / 槽宽3mm | 同上 |

**加工余量自动计算：** 系统自动比对毛坯与成品尺寸，计算各方向加工余量，在解析结果页面顶部用绿色信息条展示（如"毛坯D50→成品D45，单边余量2.5mm | 毛坯L155→成品L150，端面余量5mm"）。

**图纸预览：**
- 解析结果页面顶部并排展示两张图纸的缩略图（毛坯 + 成品）
- 点击任一缩略图可全屏放大查看原图
- 方便用户在确认解析结果时对照原图核对

**逐项编辑：**
- 每项解析结果右侧提供「✎ 编辑」按钮
- 点击后该项变为输入框，可直接修改AI解析的值
- 编辑后置信度标记保留（提示用户该项AI原本的把握度）
- 低置信度项（<60%）必须人工确认或修改后方可进入下一步

**用户确认界面：**
- 毛坯和成品分两张解析卡片展示，各有独立的标题和已提取项数标签
- 每张卡片内按置信度分级展示各项，低置信度项高亮标红
- 所有低置信度项必须用户逐项确认或修正后，才能点击「确认无误，生成G代码」

##### 子功能 3C：后处理模板化

这是解决"坑一"的核心设计。

**机床档案表**（每行一个型号）：

| 品牌 | 型号 | 控制系统 | 支持循环 | G 代码方言特征 |
|------|------|----------|----------|---------------|
| FANUC | 0i-TF | FANUC Series 0i-F | G71/G72/G70/G76/G83 | 标准 ISO，A/B/C 三类宏 |
| FANUC | 0i-MF | FANUC Series 0i-F | G81/G83/G84/G76 | 同上 |
| Siemens | 828D | SINUMERIK Operate | CYCLE95/CYCLE97/CYCLE81 | ShopTurn 风格 |
| Haas | ST-20 | Haas NGC | G71/G72/G70 | 简化 G 代码，部分代码与 FANUC 兼容 |
| GSK | 980TDi | 广数 980TDi | G71/G72/G70/G76 | 类 FANUC，但有方言差异 |

**注入机制**：用户选择机床后，系统从机床档案表读取后处理配置，注入到 AI Prompt 中作为约束条件。

**一期覆盖**：FANUC 0i-TF、FANUC 0i-MF、Siemens 828D、Haas ST-20、GSK 980TDi。

##### 子功能 3D：G 代码安全校验引擎

这是解决"坑二"的核心设计。**独立于 AI 的规则引擎**，逐行解析 G 代码 + 状态机 + 规则匹配。

**A 类 — 阻断级（必须回退重生成）：**

| 规则编号 | 规则描述 | 严重性 |
|----------|----------|--------|
| A-01 | 缺少工件坐标系声明（G54-G59） | 致命 — 可能撞机 |
| A-02 | 缺少单位声明（G20/G21） | 致命 — 英寸/毫米混淆 |
| A-03 | 缺少程序结束指令（M02/M30） | 致命 — 程序非正常终止 |
| A-04 | 运动指令前无刀具调用（Txx） | 致命 — 未装刀即运动 |
| A-05 | G00 缺少 G01 切换（仅有快进无切削进给） | 致命 — 快进撞工件 |
| A-06 | 缺少主轴启动指令（M03/M04） | 致命 — 主轴未转即切削 |
| A-07 | G02/G03 圆弧指令缺少 I/J/K 或 R 参数 | 致命 — 圆弧定义不完整 |
| A-08 | 语法错误（无法解析的 G 代码行） | 致命 — 无法执行 |
| A-09 | 主轴转速 S 值超过机床额定最大值（从机床档案读取） | 致命 — 超速危险 |
| A-10 | G00 定位点坐标超出机床行程限位（从机床档案读取） | 致命 — 超行程撞机 |
| A-11 | 子程序/宏程序调用（M98/M97/G65）未找到对应子程序定义 | 致命 — 调用未定义程序段 |
| A-12 | 刀具长度补偿（G43/G44）启用但未设置 H 值 | 致命 — 补偿量未知 |

**B 类 — 警告级（用户必须确认才能复制代码）：**

| 规则编号 | 规则描述 | 风险 |
|----------|----------|------|
| B-01 | 换刀前缺少主轴停止 M05 | 旋转中换刀，可能损坏刀架 |
| B-02 | G00 定位点坐标在毛坯范围内 | 可能快进撞工件 |
| B-03 | 车螺纹前缺少恒转速 G97 | 变螺距导致螺纹报废 |
| B-04 | 刀补激活顺序不当 | 可能过切或留余量 |
| B-05 | 缺少冷却液开关指令 M08/M09 | 干切导致烧刀 |
| B-06 | 安全换刀点不在机床限位内 | 可能超程 |
| B-07 | 同一刀具多次调用但未重新对刀 | 累计误差导致超差 |
| B-08 | 冷却液类型与材料不匹配（如铸铁干切需 M08 但未开启） | 刀具寿命缩短 |
| B-09 | G01 进给速度 F 值超过机床额定最大进给 | 可能导致步进电机丢步 |

**C 类 — 提醒级（代码中标黄）：**

| 规则编号 | 规则描述 | 标记方式 |
|----------|----------|----------|
| C-01 | 进给量 > 0.3 mm/r（粗车） | 黄色高亮 + 标注"偏大" |
| C-02 | 切削深度 > 4 mm（单刀） | 黄色高亮 + 标注"检查机床功率" |
| C-03 | 转速超过材料推荐上限 | 黄色高亮 + 标注"可能烧刀" |
| C-04 | 切削速度低于推荐范围 | 黄色高亮 + 标注"效率较低" |
| C-05 | 精加工余量不合理（过大或过小） | 黄色高亮 + 标注"检查余量" |
| C-06 | 相同刀具连续使用超过预设寿命阈值 | 黄色高亮 + 标注"建议检查刀具磨损" |
| C-07 | G00 与工件轮廓距离 < 1mm | 黄色高亮 + 标注"安全间隙偏小" |

**重试机制**：
- A 类错误 → 将错误详情注入 AI Prompt，回退重生成，最多 2 次
- 2 次后仍不通过 → 展示最后一次生成代码 + 错误详情 + 建议用户手动修改
- 每条规则必须有配套单元测试（positive + negative），在规则引擎模块通过 CI 自动执行

**设计原则：** 规则引擎不依赖 AI，不经过 LLM，确定性执行。校验引擎应独立容器化部署，输入端和输出端各过一次校验。规则不应硬编码在 Python 代码里，每条规则需结构化存储（rule_id/version/effective_date/matcher/message_template），支持运行时刷新无需重启服务。

##### 子功能 3E：装夹方案建议

这是解决"坑三"的核心设计。AI 生成 G 代码的同时，根据零件类型输出装夹方案。

| 零件类型 | 推荐装夹方式 | 关键控制点 |
|----------|-------------|-----------|
| 轴类（L/D > 3） | 三爪卡盘 + 顶尖 | 跳动 ≤ 0.02mm，检查卡爪磨损 |
| 盘类（D/T > 4） | 软爪 + 定位面 | 端面跳动 ≤ 0.01mm，软爪自镗一刀 |
| 套类 | 三爪 + 心轴/涨套 | 内外圆同轴度控制 |
| 细长轴（L/D > 10） | 跟刀架 + 中心架 | 切削力控制，防弯曲 |

##### 子功能 3F：刀路轨迹模拟

前端 Canvas 绘制简易二维刀路轨迹图：

| 元素 | 颜色 | 含义 |
|------|------|------|
| 快进轨迹 | **红色** | G00 快速定位 |
| 切削进给 | **蓝色** | G01/G02/G03 切削运动 |
| 工件轮廓 | **灰色虚线** | 毛坯/成品轮廓参考 |
| 换刀点 | **绿色圆点** | 安全换刀位置 |

目的：让用户可以**直观检查**刀路是否在工件范围内、有没有异常轨迹（如刀路穿过了工件）。

##### 子功能 3G：四层注释

生成的 G 代码自带四层注释：

```
层1：程序头信息 — O0001（零件名称）、材料、毛坯尺寸、机床型号、生成时间
层2：刀具表 — T0101 外圆车刀 DCLNR2525M12 / T0202 切槽刀 ...
层3：工序分段 — === 工序1：粗车外圆 === / === 工序2：精车外圆 === / === 工序3：切槽 ===
层4：关键指令解释 — G71 U2 R1 / (粗车循环：每次切深2mm，退刀1mm)
```

##### 子功能 3H：输出前风险确认

G 代码输出前，显示：
- **B 类警告清单**：用户必须逐项勾选确认
- **免责声明**：此代码由 AI 生成，请先在机床上空运行验证
- **空运行提示**：醒目提示"首次运行请使用空运行模式 + 单段执行"

##### 子功能 3I：材质与机床后台维护策略

**材质选择：**
- 操作端为纯下拉单选，不支持用户自定义输入
- 可选材质列表由后台管理员在"知识库管理 → 材料库"中统一维护
- 下拉框底部显示提示："材质由后台统一维护，如需新增请联系管理员"
- 材质数据包含：材料名称、类别、硬度HRC、推荐切削速度范围、进给范围、冷却要求

**机床品牌型号选择：**
- 操作端为纯下拉单选，不支持用户自定义输入
- 可选项由后台管理员在"机床档案"中统一维护
- 每台机床记录：品牌、系列、型号、数控系统、机床类型、后处理配置JSON
- 下拉框底部显示提示："机床由后台统一维护，如需新增请联系管理员"

**设计理由：** 材质和机床参数直接影响切削参数推荐和G代码生成质量。开放自定义输入会导致数据不一致（如同一材料多个名称）、参数缺失（用户只填名称不填特性），最终降低AI生成代码的可靠性。由后台集中维护确保数据一致性。

##### 子功能 3J：首件检验引导

G代码生成并试切出第一件后，操作工面临的核心问题：**怎么检？检哪些？用什么量具？**

**功能：** G代码复制后，流程不结束。页面底部出现「开始首件检验」入口，系统根据图纸解析结果自动生成检验清单。

| 检验项 | 生成规则 |
|--------|----------|
| 有公差的尺寸（φ45 ±0.02） | 自动列出，标注标准范围，推荐量具（千分尺/卡尺/高度规） |
| 粗糙度要求 | 标注Ra值和推荐检测方法（样块比对/粗糙度仪） |
| 螺纹等特征 | 列出规格，标注检具（螺纹规/环规） |

**交互：**
- 每项检验旁有「不会量？问AI」链接，点击跳转到对话问答Tab，自动带入当前工件上下文
- 操作工填入实测值后，系统自动判定合格/不合格
- 不合格项标红，AI给出可能原因（刀具磨损？参数不当？装夹变形？）
- 首件全部通过后，系统自动生成加工记录

**给老板的价值：** "上月首件合格率73%，用首件检验引导后提升到94%。省下的废品就是你的利润。"

##### 子功能 3K：刀具批次追踪

**问题：** 老师傅听声音、看切屑颜色判断何时换刀。大学生不知道，要么换太早浪费钱，要么换太晚废活。

**功能：**
- G代码生成时自动解析出程序中的刀具（T01/T02/T03...）
- 操作工录入"本批加工XX件"，系统按批次累加每把刀的加工件数
- 预设刀具寿命阈值（如CNMG120408粗车45钢≈300件），快到期时提醒
- 到期提醒："T02精车刀已切250/250件，建议下批前更换"
- 异常报废记录：刀只切了20件就崩了 → 记录并AI分析原因

**按批次追踪（非逐件）：** 操作工不需要每切一件点"+1"。开始一批时录入批次数，切完整批后系统自动更新计数。

##### 子功能 3L：加工记录自动归档

**问题：** 老师傅脑子里有"上次怎么干的"经验库，大学生没有。下次遇到类似活，一切从零开始。

**功能：**
- 首件检验通过后，系统自动生成一条加工记录，包含：零件类型、材料、机床、刀具、参数、结果、日期
- 下次上传类似图纸时，AI在解析阶段提示："根据加工记录，2026-05-12加工过类似304不锈钢轴，精车转速S220效果良好"
- 后台可查看全部加工记录、按零件/材料/机床筛选、导出报表

**长期价值：** 用得越久，数据越多，推荐越准，工厂越离不开——这就是护城河。

## 5. 三大关键坑与解决方案（综述）

### 坑一：后处理 — 数控系统方言差异

| 项目 | 内容 |
|------|------|
| **问题** | FANUC 用 G71 粗车循环，Siemens 用 CYCLE95，Haas 有简化语法，同一段刀路无法跨系统通用 |
| **方案** | 机床档案表维护"品牌 + 型号 → 后处理模板"映射；用户选择机床后自动注入 AI Prompt |
| **一期覆盖** | FANUC 0i-TF、FANUC 0i-MF、Siemens 828D、Haas ST-20、GSK 980TDi |
| **扩展性** | 机床档案表可动态扩展，后续支持客户定制后处理模板 |

### 坑二：安全校验 — 致命错误拦截

| 项目 | 内容 |
|------|------|
| **问题** | AI 可能忘记抬刀、漏设坐标系、搞错转速单位，导致撞刀废料甚至伤人 |
| **方案** | 独立规则引擎（非 AI），逐行解析 G 代码 + 状态机 + 规则匹配；A 类阻断（12 条规则）、B 类警告（9 条规则）、C 类提醒（7 条规则），共 28 条；最多重试 2 次。详见 4.3.2-3D |
| **设计原则** | 规则引擎不依赖 AI，不经过 LLM，确定性执行。校验引擎应独立容器化部署，输入端和输出端各过一次校验 |
| **法律与流程配套** | L3 用户确认（免责声明 + 空运行提示）+ L4 管理审批（3 年经验操作工复核 + 电子签名锁定 + 操作日志审计）。详见第 6 章四层安全体系 |

### 坑三：装夹和找正 — 程序的另一半

| 项目 | 内容 |
|------|------|
| **问题** | G 代码写得再好，装夹错了全白给。大学生不知道怎么装夹 |
| **方案** | AI 生成代码同步输出装夹方案文本；前端 Canvas 绘制刀路轨迹图（红/蓝/灰三色）；用户可直观检查刀路是否在工件范围内 |
| **关键价值** | 把隐性的装夹知识显性化输出，降低操机风险 |

## 6. 安全体系（四层防线）

```
┌─────────────────────────────────────────┐
│  第一层：图纸质量门禁                      │
│  本地实时质检 + 服务端精细校验              │
│  ↓ 通过                                  │
│  第二层：G 代码静态校验引擎                  │
│  A 类阻断 → 回退重生成 / B 类警告 / C 类提醒 │
│  A 类 12 条 + B 类 9 条 + C 类 7 条       │
│  ↓ 通过                                  │
│  第三层：用户确认护栏                        │
│  B 类强制勾选 / 免责声明 / 空运行提示         │
│  ↓ 通过                                  │
│  第四层：管理审批层                          │
│  3年以上经验操作工复核 / 电子签名锁定 / 操     │
│  作日志审计                                 │
└─────────────────────────────────────────┘
```

| 层级 | 安检内容 | 失败处理 |
|------|----------|----------|
| L1 — 图纸门禁 | 分辨率、清晰度、倾斜角度、文件类型（本地 + 服务端双检） | 本地失败→拍照预览提示重拍（2步内反馈）；服务端失败→提示重新上传 |
| L2 — 代码校验 | A/B/C 三级规则（共 28 条，详见 4.3.2-3D） | A 类回退重生成（最多 2 次）；B/C 类展示警告 |
| L3 — 用户确认 | B 类警告逐项勾选、免责声明阅读确认、空运行提示 | 未全部勾选则「复制代码」按钮不可用 |
| L4 — 管理审批 | G代码输出后可选"发送审核"，3年以上经验操作工复核审查代码，审核通过带电子签名锁定 | 未审批代码建议"先空运行验证后再上机" |

## 7. 技术约束

| 约束项 | 内容 |

### 7.1 前端选型决策记录

**问题：** MUI + Tailwind CSS 双轨并行会导致 CSS Specificity 冲突、两套主题 token 需同步维护、bundle 体积叠加。需在 MVP 开发前二选一。

**选择方案对比：**

| 维度 | MUI (Material UI) | Tailwind + Radix UI |
|------|-------------------|---------------------|
| 组件生态 | 完整（表单/表格/弹窗/导航全量覆盖） | 基础组件（Radix 提供 headless 原语） |
| 定制灵活性 | 需通过 ThemeProvider + sx 属性 | 极高（utility-first） |
| Bundle 体积 | ~100KB gzip（含图标） | ~30KB gzip（Tailwind + Radix） |
| 入门成本 | 中等（需学习 MUI 组件 API） | 低 |
| 适用场景 | 原型快速迭代、内部工具类 | 高度定制化、C 端精致体验 |

**推荐：** 初期团队若 React 经验为主 → **选 MUI**（组件生态完整，快速搭建原型）。若产品定位为高度定制化 C 端体验 → **选 Tailwind + Radix UI**。选定后删除另一依赖，CSS tokens 统一到一套系统。

**文件类型说明：** PRD 前端标签页（第 1 章"编程语言"）保留两者，但技术评审会应做出具体决策。

### 7.2 离线与弱网策略

> **工厂场景：** 车间地下室、钢结构厂房内部信号极差。全文多处假设在线（AI 问答、RAG 检索、图纸上传），但不覆盖离线/弱网情况。

**MVP 离线策略：**

| 能力 | 实现方式 | 优先级 |
|------|----------|--------|
| 核心 Shell 缓存 | Service Worker 缓存 Web App 静态资源（HTML/CSS/JS） | P0 |
| 最近 G 代码缓存 | IndexedDB 缓存最近 10 条 G 代码程序及校验结果，离线可查看/复制 | P0 |
| 安全校验引擎本地可用 | 校验引擎编译为 WebAssembly 在前端独立运行，不依赖网络 | P0 |
| 拍照上传队列重传 | 拍摄后先存入本地队列，网络恢复后自动上传 | P1 |
| 常用知识库离线索引 | 离线缓存常见报警码和切削参数（约 5MB） | P2 |

### 7.3 AI Provider 抽象

```python
class AIProvider(ABC):
    @abstractmethod
    def chat(self, messages: list, system: str | None = None) -> str: ...
    @abstractmethod
    def vision_analyze(self, image: bytes, prompt: str) -> VisionResult: ...

class DeepSeekProvider(AIProvider):
    """线上 DeepSeek API"""

class LocalProvider(AIProvider):
    """本地 ollama/Qwen，支持离线"""

class FallbackProvider(AIProvider):
    """主 Provider 失败时自动切换到备用"""
```

**目的：** 避免 DeepSeek 单一依赖，支持离线/内网降级。安全校验引擎在任何 Provider 下独立工作（不依赖 AI 模块和数据库）。

## 11. 竞品分析

### 11.1 市场竞争格局

| 竞品类别 | 代表产品 | 优势 | 局限 | 本产品差异化 |
|----------|----------|------|------|-------------|
| **传统 CAM 软件** | UG/NX、Mastercam、Edgecam | 功能完整、工业验证、支持五轴 | 学习周期长（3-6 个月）、价格高（¥5 万+）、桌面端 | 移动端优先、零学习成本、AI 驱动、即时出代码 |
| **在线 G 代码生成器** | CNC G Code Generator、ncgenerator.com | 免费/廉价、浏览器可用 | 无机型适配、无安全校验、无知识库 | 安全校验引擎 + 后处理模板 + RAG 知识库 + 首检追踪 |
| **数控仿真软件** | Vericut、NC Simul | 高精度物理仿真、碰撞检测 | 重型工业软件、不适合日常编程 | 轻量级刀路预览、AI 诊断替代完整仿真 |
| **工艺手册/切削参数 App** | 机械加工工艺手册、切削参数计算器 | 参数直接可用 | 仅限参数查询、无编程/排故能力 | G 代码生成 + 故障诊断 + 首检一体化 |
| **老师傅经验传承** | 工厂内部师徒制 | 现场适应性强、经验丰富 | 培训周期长（3-6 月）、高度依赖个人 | AI 编码老师傅经验、7×24 可用、不离职 |

### 11.2 差异化定位总结

```
传统 CAM  → 重、贵、慢     → 适合编程工程师
在线工具  → 轻但弱（无安全） → 适合临时需求
本产品    → 手机端 + AI + 安全校验引擎 + 知识库
            → "大学生 + AI = 老师傅七成功力"
```

### 11.3 护城河

1. **安全校验引擎** — 独立于 AI 的确定性执行，行业内未见到类似产品将安全校验做在产品层面
2. **后处理模板库** — 每新增一个机床品牌/型号的模板，就增加一份网络效应
3. **加工记录数据** — 用得越久，历史加工数据越丰富，AI 推荐越精准（第 3K 节）
4. **首件检验引导** — 将"怎么检、用什么量具、合格标准"结构化，降低新人检验门槛

| 元素 | 触发方式 | 行为 |

| 「⚠️ 程序有问题？反馈修正」按钮 | 点击 | 弹窗列出问题类型（尺寸超差/表面质量/刀具异常/其他），选类型并描述现象→AI基于原参数微调重新生成G代码。图纸和参数全部保留不重新上传 |

| 已保存档案列表 | 自动 | 按时间倒序排列，最新保存的置顶。每条显示零件名、材质、机床、时间、刀具配置 |
